常见Hook方式原理对比

1. IAT Hook（导入表劫持）

  • 原理：修改PE文件的导入地址表（IAT），将API函数指针替换为自定义函数地址。
  • 流程：定位IAT表 → 找到目标API指针 → 修改为自定义函数地址。
  • 优点：实现简单，适合API劫持，稳定。
  • 缺点：仅能hook导入API，易被检测，无法hook内部函数。

2. Inline Hook（内联劫持）

  • 原理：直接修改目标函数机器码开头，写入jmp跳转到自定义代码（通常为5字节jmp rel32或12字节jmp abs）。
  • 流程：保存原始指令 → 修改入口为jmp → 自定义代码执行后可跳回原函数剩余部分（trampoline）。
  • 优点：可hook任意函数，灵活，常用于逆向和CTF。
  • 缺点：需处理指令对齐、trampoline，易被检测，破坏原始代码段。

3. Code Overwrite Hook（代码覆盖劫持）

  • 原理：覆盖任意地址的机器码为jmp或call，劫持执行流到自定义代码。
  • 流程：定位目标地址 → 保存原始指令 → 覆盖为jmp/call → 自定义代码执行后可跳回。
  • 优点：可hook任意代码位置，极为灵活，适合逆向分析、破解。
  • 缺点：破坏性强，需手动处理指令边界，易引发崩溃。

4. Frida Hook

  • 原理：基于Frida动态插桩框架，通过注入JS脚本，动态修改目标进程内存、函数实现或API行为。
  • 流程：Frida注入目标进程 → JS脚本定位并hook函数 → 动态修改参数/返回值/行为。
  • 优点：跨平台、无需重启进程、支持动态分析、逆向、自动化。
  • 缺点：需依赖Frida环境，易被反检测，hook粒度受限于Frida API。

5. Ptrace Hook

  • 原理：利用Linux的ptrace系统调用，父进程控制/调试子进程，动态修改其内存、寄存器、代码，实现hook。
  • 流程：父进程ptrace attach目标 → 读写内存/寄存器 → 注入shellcode或修改指令 → 实现劫持。
  • 优点：无需修改目标文件，适合动态调试、逆向、自动化分析。
  • 缺点：仅限Linux，操作复杂，性能开销大，易被检测。

总结对比表

方式 典型场景 优点 缺点
IAT Hook API劫持 稳定，易实现 仅API，易检测
Inline Hook 任意函数/逆向 灵活，常用 破坏性，需trampoline
CodeOverwrite 任意代码/破解 极灵活，任意地址 破坏性强，易崩溃
Frida Hook 动态分析/自动化 跨平台，动态，强大 依赖环境，易反检测
Ptrace Hook Linux调试/逆向 无需改文件，动态 仅Linux，复杂

实际逆向和CTF中，常结合多种hook手段，选择最适合场景的方式。

原理

IAT表存储着当前这个模块所用到的所有api函数地址，并且这些api函数在IAT表里的位置都是挨在一起的，因此只要将这些函数的地址修改成指向我们自行设置的函数地址，就能实现所谓的IAT Hook

具体操作原理是通过GetModuleHandle读取该exe加载到内存中的地址，因为Dos中存放PE头的偏移量，IAT表偏移量存放在PE中，因此我们通过访问PE中的OptionalHeader中的DataDirectory，就能读取出import表偏移量，通过一开始的加载到内存中的地址+偏移量 == 最终地址，就可以找到所调用的函数、系统调用、API等。因为只可读，所以得通过VirtualProtect实现改权限，更改指向地方

Code Overwrite Hook

基于汇编层面的手动劫持任意地址执行流的 Hook 技术，关键点是使用 汇编重写目标函数开头的指令，并跳转到我们自己的代码段

比较上InlineHook，也采用jmp，但采用BeaEngine(轻量级反汇编引擎)

Hook方式 本质手段 可Hook对象 修改点 原函数调用保留 特点
IAT Hook 修改导入地址表（Import Table） 仅限 API 函数 .idata 段 IAT 表 ✅ 一般无需 trampoline 稳定、易检测，常用于 API 劫持
Inline Hook 在函数入口写 jmp 跳转 通常是函数 函数入口前几字节 ✅ trampoline 保存原始指令 常用于函数间跳转，执行完可以回调原函数
Code Overwrite 覆盖任意地址代码并劫持执行流 函数/任意地址（代码段） 任何代码地址 ❌ 不一定保留原逻辑，依赖场景 灵活，可用于关键指令覆盖，偏逆向分析

Ptrace Hook

ptrace一般用于双线程或frida等其他中

ptrace：Linux 内核提供的系统调用，用于让一个进程控制另一个进程的执行
long ptrace(enum __ptrace_request request, pid_t pid, void *addr, void *data);
其中request有

request 值 含义
PTRACE_TRACEME 当前进程表示“我希望被调试”，通常子进程中调用（用于 gdb 这类调试器）
PTRACE_ATTACH 附加到一个已存在的进程，开始调试它（会暂停它）
PTRACE_DETACH 脱离调试，让目标进程恢复正常执行
PTRACE_CONT 继续执行目标进程（从上一次暂停的地方）
PTRACE_SINGLESTEP 单步执行一条指令，然后暂停（调试器常用）
PTRACE_PEEKDATA 从目标进程地址空间中读取数据（一次读取一个 long）
PTRACE_POKEDATA 向目标进程地址空间中写入数据
PTRACE_GETREGS / PTRACE_SETREGS 读取/设置通用寄存器
PTRACE_SYSCALL 让目标进程在每次系统调用进入/退出时都暂停（可以做系统调用监控）

PID：PTRACE_TRACEME为0，其他为要附加的PID，PTRACE_TRACEME经常用于父子进程中

addr：对于PEEKDATA、POKEDATA为读写的内存地址，其他一般为0

data：含义也取决于 request：

request 类型 data 的含义
POKEDATA 要写入目标地址的数据
PEEKDATA 结果通过返回值获得，data 通常为 0
CONT 如果进程因为信号暂停，你可以在这里传递信号号（如 SIGSTOP、0 表示无）
GETREGS 指向 struct user_regs_struct 的指针，用于接收寄存器内容
SETREGS 同上，传入要设置的新寄存器值
